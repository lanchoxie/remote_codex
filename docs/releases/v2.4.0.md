# v2.4.0 Recovery Checkpoint

Released: 2026-06-26

`v2.4.0` is a recovery checkpoint after the Windows desktop, API profile, history, search, export, and local-agent reliability work since `v2.3.0`. It is intended to be a stable restore point before further debugging of Windows session resume/start latency.

## Highlights

- Added Windows desktop usage documentation and screenshots for desktop sessions, host management, API profiles, export options, and history attachments.
- Added API profile management with default key persistence, host-level API mappings, API ping checks, and restart prompts when live sessions are using stale API mappings.
- Expanded session discovery and history import so Codex rollout JSONL files can be tailed and imported into remote sessions.
- Preserved full transcript/thinking history on first session open, while keeping optimized session-list payloads for navigation performance.
- Added full-history search across cached transcript and rollout files, with keyword highlighting plus previous/next navigation inside a conversation.
- Added richer export options for Markdown, JSON, and ZIP bundles, including thinking/activity, images, files, extension filters, date ranges, multi-date selection, and select-all dates.
- Improved file handling with online preview, download/open/save actions, inline image previews, and large/chunked file transfer support.
- Added local Windows agent watchdog and restart/stop controls, plus more detailed managed-session startup diagnostics.
- Added `runId` tracking for managed session launches to reduce stale start/resume events polluting newer sessions.
- Added native Codex helpers for model listing, skill listing, Goal controls, Plan fallback, and app-server lifecycle diagnostics.
- Added frontend performance controls including optimized session lists, long transcript window rendering, image preview limits, and reduced unnecessary rerenders.

## Important Behavior Changes

- Managed Codex app-server sessions now use isolated `.remote-codex-managed/.../.codex` homes for API-profile/runtime separation.
- Running app-server sessions keep the API environment they started with; changing an API key or Base URL requires restarting affected managed sessions.
- History/detail APIs distinguish optimized list payloads from full transcript/detail requests.
- Local agent command queues now include acknowledgement, retry/timeout handling, and transient network error handling.

## Known Follow-Up Areas

- Windows session resume/start can still be slower than HPC/Linux in some environments. Continue debugging by checking app-server startup phases, `.remote-codex-managed`, SQLite state, relay/agent heartbeat, and command queue timing.
- Multi-window/multi-session workflows need a dedicated architecture pass before implementation.
- GitHub Release notes should mirror this file when creating the public release page.
